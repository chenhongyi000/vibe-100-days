还在切到手机看天气？一行命令，终端里直接查

Vibe Coding 100 天挑战 · Day 3

01 一个每天都在发生的"上下文切换"

你有没有过这样的经历——

正在写代码，写到一半，想看一眼外面天气怎么样，要不要带伞。于是你：

  1. 放下键盘
  2. 拿起手机
  3. 解锁屏幕
  4. 找到天气 App
  5. 点开、等加载
  6. 看一眼
  7. 放下手机
  8. 重新盯着屏幕，想"刚才我写到哪了？"

整个过程 30 秒。但真正让你付出的，不是这 30 秒，而是被打断的心流。

研究说，一次中断后，重新进入深度专注状态平均需要 23 分钟。你每天看几次天气？三次？四次？每次 30 秒，换来的是 23 分钟 × 4 = 一个半小时的"假性工作时间"。

所以我做了 Day 3 的项目——

wttr-cli，终端天气极简仪。

一行命令，看一眼天气，然后继续写代码。

02 为什么是"极简"？

市面上不是没有天气工具。手机有 App，浏览器有网页，甚至桌面有小部件。但它们的共同问题是：需要你离开终端。

对开发者来说，终端就是宇宙中心。写代码在这、调试在这、Git 在这、装包在这——为什么要为看个天气离开？

所以我给 wttr-cli 定下了三个设计原则：

1. 零配置

不用注册、不用申请 API Key、不用填任何表单。装完就用。这是底线。任何一个需要"先去某网站注册"的工具，在开发者眼里都是一道坎。

2. 一行命令

wttr 北京

就这。没有二级菜单，没有交互式问答，没有多余的选项。输入城市名，回车，天气就出来了。不输入城市名？自动检测你的位置。

3. 美，但实用

终端输出可以很丑，也可以很美。我选了 Rich 库来做渲染——彩色面板、表格、温度颜色编码。红色就是热，蓝色就是冷，一眼看过去就知道今天穿什么，不需要读数字。

03 技术实现：三个关键决策

整个工具核心代码不到 300 行，但在做之前有三个关键决策要定。

决策一：天气数据从哪来？

选项有这么几个：

数据源 需要 API Key 支持中文 免费额度 返回格式
OpenWeatherMap ✅ 需要 ❌ 1000次/天 JSON
和风天气 ✅ 需要 ✅ 1000次/天 JSON
中国气象局 ✅ 需要 ✅ 有限 JSON
wttr.in ❌ 不需要 ✅ 无限 JSON/ANSI

我选了 wttr.in。

理由很简单：它是唯一一个真正做到"零配置"的选项。不需要注册、不需要 API Key、不需要申请、不需要等审批。你打开浏览器访问 wttr.in/Beijing 就能看到天气。它支持中文城市名（wttr.in/北京），返回 JSON 格式（?format=j1），全球覆盖。

wttr.in 是一个社区维护的免费服务，底层用的是 World Weather Online 的数据。它存在的意义就是"让天气查询变得简单"——和 wttr-cli 的理念完全一致。

决策二：怎么让终端输出"好看"？

终端美化有三种思路：

  1. 手动拼接 ANSI 转义码 — 灵活但繁琐，跨平台兼容性差
  2. 用 Python 的 colorama 库 — 解决了跨平台问题，但只能着色
  3. 用 Rich 库 — 面板、表格、树、进度条、Markdown 渲染，全平台支持

我选了 Rich。原因很简单：它提供的 Panel 和 Table 组件，恰好就是"天气卡片"和"预报表格"的天然载体。

# 当前天气用 Panel 包裹，像一个卡片
panel = Panel(content, title=f"[bold]{city_name}[/bold]",
              border_style="cyan")

# 预报用 Table 展示，像数据表格
table = Table(title="天气预报", box=box.SIMPLE_HEAVY)
table.add_column("日期")
table.add_column("天气")
table.add_column("高温/低温")
# ...

温度的颜色编码也很有意思。我没有用随机颜色，而是让温度本身决定颜色：

def _temp_color(temp_c: int) -> str:
    if temp_c >= 35:   return "bold red"    # 酷热
    elif temp_c >= 30: return "red"         # 热
    elif temp_c >= 20: return "yellow"      # 温暖
    elif temp_c >= 10: return "green"       # 舒适
    elif temp_c >= 0:  return "cyan"        # 凉爽
    else:              return "bold blue"   # 冷

这样你不需要读数字，扫一眼颜色就知道今天大概什么温度。红色警报，蓝色加衣，绿色放心。

决策三：天气代码怎么映射？

wttr.in 使用的是 WWO（World Weather Online）天气代码，不是 WMO 标准。这意味着不能直接用"查表"的方式来获取中文描述。

我在代码里维护了一个 49 个代码的映射表：

WEATHER_CODES = {
    "113": ("☀️", "晴"),
    "116": ("⛅", "多云"),
    "119": ("☁️", "阴"),
    "263": ("🌧️", "小雨"),
    "299": ("🌧️", "中雨"),
    "305": ("🌧️", "大雨"),
    "359": ("🌧️", "暴雨"),
    "200": ("⛈️", "雷阵雨"),
    "323": ("🌨️", "小雪"),
    "329": ("🌨️", "中雪"),
    "335": ("🌨️", "大雪"),
    # ... 共 49 个
}

这是整个项目里最"不极简"的部分——纯手工活，一个一个对着 WWO 的文档写的。 但这也是它和直接 curl wttr.in 的核心区别——wttr.in 只返回英文天气描述，而 wttr-cli 把每一个天气代码都翻译成了中文 + emoji。

Emoji 的选择也是刻意的。在终端里，emoji 是最节省空间的视觉信号。一个 ☀️ 比 "Sunny" 四个字符更直观，而且不依赖语言。

04 使用场景

场景一：出门前看一眼

$ wttr
# 自动检测你的位置，显示当前天气 + 3 天预报

场景二：出差前查目的地

$ wttr 上海
$ wttr "San Francisco"
# 中英文都支持

场景三：脚本集成

$ wttr --simple
# 北京: ☀️ 晴 25°C (体感 24°C) | 湿度 45% | 西南风 15 km/h

一行输出，适合放在 shell 启动脚本里，或者 tmux 状态栏，或者 cron 定时任务推送到通知。

场景四：SSH 到服务器时看一眼

$ ssh myserver
$ wttr
# 服务器的 IP 定位，直接显示当地天气
