## 01 一个所有人都经历过的尴尬场景

你有没有遇到过这样的时刻——

想跟 AI 助手（ChatGPT、Claude）聊一个项目的架构，复制粘贴整个项目不现实，只说文件名又说不清楚。于是你手动在聊天框里敲：

```
├── src/
│   ├── main.py
│   └── utils.py
└── tests/
    └── test_main.py
```

敲到第三层就开始怀疑人生——缩进对不齐，`│` 和 `├` 混在一起，一个文件名改了，下面全要重新对齐。

或者更惨的：昨天我为了给一个项目生成目录树，手动在 Markdown 里敲了 30 分钟。结果今天项目结构调整了一下，全部作废重来。那一刻我意识到：

**这种重复劳动，必须交给工具。**

## 02 为什么是"树形生成器"？

很多人可能会说：系统自带的 `tree` 命令不就行了吗？

还真不行。

`tree` 命令的输出不是 Markdown 格式，直接粘贴到文档里格式会乱。更重要的是——`tree` 会把所有东西都列出来。一个典型的 Web 项目跑一下：

```
project/
├── node_modules/
│   ├── lodash/
│   │   ├── lodash.js
│   │   └── ...  （还有 8000 个文件）
├── .git/
│   └── ...
├── __pycache__/
├── dist/
└── src/
    └── main.py   ← 你真正关心的东西在这里
```

500 行噪音，3 行有用信息。

`tree` 虽然也有 `-I` 参数可以排除，但它是简单的 glob 匹配——不支持取反规则（`!`），不支持路径相关的匹配（`/dist/` 只匹配根目录下的 dist）。

### 核心矛盾

**项目的"物理结构"和"文档中的结构描述"之间的割裂。**

项目在变，文档跟不上。手动维护目录树，注定是过时的。

所以这个工具做了三件事：

**第一，自动扫描，一键生成。** 输入一个项目路径，递归扫描所有文件和文件夹，输出一棵完整的目录树。

**第二，智能过滤——这是最大的亮点。** 它直接读取项目根目录的 `.gitignore` 文件，自动排除那些"开发者已经达成共识不需要关注"的文件。这意味着：
- 不需要手动配置排除列表
- 每个项目的排除规则天然适配
- 输出永远是一份干净的、只包含核心代码和资源的"项目地图"

就算项目没有 `.gitignore`，它也有一套内置的默认噪声列表（`node_modules/`, `__pycache__/`, `.git/`, `dist/` 等），开箱即用。

**第三，输出格式友好。** 生成的树用标准的 Unicode 树形字符（`├──`, `│`, `└──`），可以直接粘贴到 README、技术文档、甚至 AI 对话中作为"上下文背景板"。

## 03 技术实现：四个设计决策

### 决策一：为什么不用现成的 `tree` 命令？

`tree` 的定位是终端可视化，不是文档生成。它的输出是终端格式，不是 Markdown。而且 `-I` 参数只能做简单的 glob 排除，不支持 `.gitignore` 的完整语法。

我需要的不是一个"看起来像树"的工具，而是一个**既过滤又格式化**的工具。

### 决策二：如何解析 `.gitignore`？

`.gitignore` 的规则比你想象中复杂得多：

- `*.pyc` —— 通配符匹配
- `/dist/` —— 只匹配根目录下的 dist，不是子目录里的
- `**/logs/` —— 匹配任意深度的 logs 目录
- `!important.log` —— 取反规则，把之前忽略的文件重新加回来

用正则表达式硬解析？那得写几百行正则，还容易有边界情况。

**我选择了 `pathspec` 库。** 这个库是 Python 生态中 `.gitignore` 解析的标准方案，GitHub 的 `gitignore` 规则就是用它兼容的语法。它一行代码就搞定所有复杂的模式匹配：

```python
import pathspec
spec = pathspec.PathSpec.from_lines("gitignore", patterns)
spec.match_file("dist/bundle.js")  # True
spec.match_file("src/main.py")      # False
```

包括取反规则、目录专用模式（`/` 结尾）、递归通配符（`**`），全部原生支持。

实际用的时候踩了一个小坑：`pathspec` 较新版本移除了 `match_file()` 的 `is_dir` 参数，改由路径本身是否以 `/` 结尾来判断。所以在目录匹配时需要手动补上尾斜杠，否则像 `dist/` 这样的模式会匹配不到。

### 决策三：空文件夹要不要保留？

`os.walk` 会正常遍历空目录。但有些空文件夹在项目中有结构意义——比如 `logs/`、`uploads/`、`tmp/`。它们的存在本身就是一种文档："这个目录应该有内容"。

所以我的工具会保留空文件夹，在树中显示为 `logs/`（带 `/` 后缀，没有子项），让读者知道它们是有意存在的。

### 决策四：输出文件被覆盖了怎么办？

这是一个很实际的工程问题。假设你的项目里有一份手写的 `STRUCTURE.md`，里面不仅有目录树，还有你对每个模块的说明。如果工具一不小心把它覆盖了……

**默认拒绝覆盖。** 如果 `--output` 指向的文件已存在，工具会报错退出，并提示使用 `--force` 标志来强制覆盖。

写入时还用了**原子写入**模式：先写到临时文件 `.tmp`，成功后再用 `os.replace()` 替换目标文件。这样即使写到一半中断了（断电、Ctrl+C），也不会留下一份半截的文件。

## 04 应用场景

### 场景一：喂给 AI 助手

把生成的目录树复制到 ChatGPT / Claude 的对话中，AI 就能瞬间理解你的项目结构。而且因为过滤掉了 `node_modules/` 这些噪音，节省了几百个 token——省下来的 token 可以用来贴更多的实际代码。

```
这是项目结构：
├── src/
│   ├── api/
│   ├── models/
│   └── services/
├── tests/
└── config.yaml

请帮我分析一下 api/ 和 services/ 之间的依赖关系...
```

### 场景二：写技术文档 / README

开源项目、团队文档、技术博客——任何需要展示项目结构的地方，一键生成，永不过时。每次发版前跑一次，目录树永远和代码同步。

### 场景三：团队 Onboarding

新成员加入团队，发给他一份项目目录树。他能在 10 秒内定位"代码在哪"、"配置在哪"、"文档在哪"，而不是在几十个文件夹里翻来翻去。

### 场景四：项目架构评审

在 Code Review 或架构评审中，用目录树快速展示模块划分。对比两个分支的目录树 diff，一眼就能看出这次改动涉及了哪些模块。

## 05 使用方式

```bash
# 基本用法
tree-gen ./my-project

# 输出到文件
tree-gen ./src --output tree.md

# 限制深度
tree-gen ./ --depth 2

# 显示所有内容（禁用过滤）
tree-gen ./ --no-gitignore

# 强制覆盖已有文件
tree-gen ./ --output tree.md --force
```
